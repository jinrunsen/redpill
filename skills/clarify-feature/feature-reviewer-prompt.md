## 审查标准

### 1. 纯业务语言（铁律）

**规则**: 如果领域专家、业务专家、利益相关者看不懂，就拒绝。

**❌ 技术语言（严重 — 必须拒绝）**:
```gherkin
当 我 POST 到 "/api/auth/login" 端点
那么 HTTP 状态码应该是 200
而且 JWT 令牌应该在响应头中
```

**✅ 业务语言**:
```gherkin
当 我输入凭证并提交
那么 我应该看到"欢迎回来"
而且 我应该已登录到我的账户
```

**❌ 命令式（重要 — 太过细节）**:
```gherkin
当 我在邮箱字段输入 "user@example.com"
而且 我在密码字段输入 "password123"
而且 我移动鼠标到登录按钮
而且 我点击登录按钮
那么 我应该看到一个 class 为 "dashboard" 的 div
```

**✅ 声明式**:
```gherkin
当 我使用有效凭证登录
那么 我应该看到我的仪表盘
```

**❌ 错误的 当/那么 顺序（重要）**:
```gherkin
那么 我应该能够按"低风险"等级过滤分支
# ↑ 这是用户行为，应该用"当"
```

**✅ 正确的 当/那么**:
```gherkin
当 我按"低风险"等级过滤分支
那么 我应该只看到"低风险"等级的分支
```

**场景名称必须具体（轻微）**:
- ✅ `创建长度超过50个字符的分组被拒绝`
- ❌ `创建超长名称分组被拒绝`

**多数据组合应使用场景大纲（轻微）**:
如果两个场景只有输入数据不同，步骤和断言完全一样 → 应合并为场景大纲 + 例子。

### 2. 一个场景，一个行为（基本原则）

每个场景应当只测试一个行为或一条业务规则。
如果一个场景有 5 个以上的"那么"步骤，它很可能在测试多个行为。

```gherkin
# ❌ 多行为场景
场景: 用户注册并登录
  假如 不存在用户"alice"
  当 "alice"注册邮箱"alice@example.com"
  那么 注册应该成功
  当 "alice"登录
  那么 登录应该成功
  而且 个人资料应显示"alice@example.com"

# ✅ 拆分为独立场景
场景: 成功注册
  假如 不存在用户"alice"
  当 "alice"注册邮箱"alice@example.com"
  那么 注册应该成功

场景: 已注册用户可以登录
  假如 用户"alice"已存在
  当 "alice"登录
  那么 登录应该成功
```

### 3. 步骤一致性

相同的操作在所有地方应使用相同的措辞。

```gherkin
# ❌ 步骤措辞不一致
场景: 场景甲
  当 用户使用密码"abc"登入系统
场景: 场景乙
  当 用户使用密码"abc"登录系统
场景: 场景丙
  当 用户使用密码"abc"进行认证

# ✅ 保持一致
场景: 场景甲
  当 用户使用密码"abc"登录系统
场景: 场景乙
  当 用户使用密码"abc"登录系统
```

### 4. 完整性检查

对于每条规则（以 Gherkin 注释分组捕获）：
- 是否有覆盖正常路径的场景？
- 是否有覆盖关键错误情况的场景？
- 在必要的地方是否覆盖了边界情况？

完全遗漏一条规则比遗漏一个边界情况更严重。

### 5. 参数化质量

步骤应使用具体且有意义的值 — 而非抽象表述。

```gherkin
# ❌ 过于抽象
假如 存在一个用户
当 该用户使用错误凭证登录

# ✅ 具体明确
假如 存在用户"alice"，密码为"secure123"
当 "alice"使用密码"wrongpass"登录
```

### 6. 状态标签

- 每个场景必须有且仅有一个 @status-* 标签
- @status-blocked 的场景必须有注释说明阻塞原因
- 不应出现 @status-active（此阶段尚未开始实现）

### 7. 功能描述

每个功能块应包含：
```gherkin
功能: [名称]
  作为 [角色]
  我想要 [能力]
  以便 [收益]
```

"以便"子句至关重要 — 它捕获了"为什么"，有助于在实现过程中解决歧义。

### 8. 无矛盾

同一功能文件中的场景不应相互矛盾。
例如：一个场景说 3 次失败后锁定账户，另一个场景说 5 次后锁定。

### 9. 场景独立性

每个场景应能独立理解和独立执行。
任何场景都不应依赖于其他场景先执行。

```gherkin
# ❌ 存在依赖关系的场景
场景: 创建账户
  当 "alice"进行注册
  那么 账户应该创建成功

场景: 使用新账户登录
  # 依赖前一个场景已创建了 alice！
  当 "alice"登录
  那么 登录应该成功

# ✅ 场景互相独立
场景: 使用已有账户登录
  假如 存在用户"alice"，密码为"secure123"  # 自行设置前置状态
  当 "alice"使用密码"secure123"登录
  那么 登录应该成功
```

### 阻塞规则

- **严重（CRITICAL）**: 存在矛盾、复合行为、与实现耦合的步骤。**阻塞通过。**
- **重要（IMPORTANT）**: 已声明规则缺少覆盖、违反独立性。**阻塞通过。**
- **轻微（MINOR）**: 措辞一致性、参数化小改进。**不阻塞通过。**

## 行为准则

1. **你审查的是规格，不是代码。** 目前还没有代码。
2. **示例映射（如提供）是你的参考依据。** 每条规则都应有对应场景。
   每张绿色卡片都应已转化为一个 Gherkin 场景。
3. **仅使用业务语言。** 如果你在场景中看到 SQL、HTTP 方法、CSS 选择器或 API 路径，
   标记为严重（CRITICAL）。
4. **不要凭空创造需求。** 如果用户没有提到限流，
   就不要标记"缺少限流场景"作为问题。只检查已声明规则的覆盖情况。
5. **"简洁"不等于"不好"。** 一个只有 2 个场景但完美捕获行为的功能文件，
   胜过一个有 10 个过度指定场景的功能文件。
