# 澄清功能

> 在任何实现之前使用。通过协作对话将原始想法精炼为结构良好的 .feature 文件。
> 此技能的产出就是规格说明本身。
> 每个项目都需要经过这个过程 -- 即使是"简单"的项目也不例外。
> 如果你是被作为子代理调度的，请跳过此技能。

---

## 核心原则

**在 BDD 中，设计产物就是 .feature 文件。**

我们使用 **Example Mapping**（示例映射）-- 一种来自 BDD 社区的结构化发现技术
-- 在编写场景之前探索问题空间。

---

## 触发条件

- 人类描述了想要构建的东西
- 人类说"定义这个功能"或"帮我设计这个"
- 尚无 .feature 文件
- 已有 .feature 文件需要新增场景

---

## 流程概览

```
1. 探索项目上下文
2. 识别用户故事
3. 对每个故事进行 Example Mapping（发现 规则 → 示例 → 问题）
4. 需求完整性检查
5. 将示例转换为 Gherkin
6. 展示并获得确认
7. 写入 .feature 文件
8. 功能审查（子代理）
9. 提交
```

---

## 第一阶段：探索项目上下文

在提出任何问题之前，先了解已有的内容：

1. **阅读项目文件**：AGENTS.md、CLAUDE.md、README 等
2. **检查已有的 .feature 文件**：我们是在扩展一个已有的 BDD 项目吗？
3. **检查已有的测试**：是否有测试应该转化为场景？
4. **了解技术栈**

总结你的理解，并与人类确认。

---

## 第二阶段：识别用户故事

通过对话提取用户故事。每次只问一个问题。

```
作为 [角色]
我想要 [能力]
以便 [收益]
```

引导人类写出符合以下标准的故事：
- **独立的** -- 可以单独构建和测试
- **有价值的** -- 为用户或业务交付价值
- **小的** -- 一次迭代内可完成
- **可测试的** -- 可以用具体示例验证

如果需求较大，建议分解。

---

## 第三阶段：Example Mapping（逐故事进行）

为每个用户故事构建一张 Example Map（示例映射图）：

```
┌─────────────────────────────────────────────────────────┐
│  STORY（故事）                                            │
│  作为注册用户，我想要登录系统                                │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  RULE（规则）: 有效凭证允许访问                              │
│  ├── EXAMPLE（示例）: Alice 使用正确密码登录                 │
│  └── EXAMPLE（示例）: Bob 使用正确密码登录                   │
│                                                          │
│  RULE（规则）: 无效凭证被拒绝                                │
│  ├── EXAMPLE（示例）: 错误密码 → 错误提示                    │
│  ├── EXAMPLE（示例）: 未知用户名 → 相同错误提示               │
│  └── EXAMPLE（示例）: 空密码 → 验证错误                      │
│                                                          │
│  RULE（规则）: 多次失败后锁定账户                             │
│  ├── EXAMPLE（示例）: 失败5次 → 锁定15分钟                   │
│  ├── EXAMPLE（示例）: 失败4次然后成功 → 计数器重置             │
│  └── QUESTION（问题）: 锁定到期后会发生什么？                 │
│                                                          │
│  QUESTION（问题）: 是否还需要按 IP 进行速率限制？             │
│  新故事: 密码重置（推迟处理）                                 │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 如何引导

**发现 RULE（规则）**：逐个询问约束条件和验收标准。
**发现 EXAMPLE（示例）**：对每条规则，询问具体案例（"就是那个……的情况"）。
**浮现 QUESTION（问题）**：遇到不清楚的地方，记录下来而非假设。
**发现新故事**：如果某条规则属于不同的功能范围，将其拆分出去。

### 健康映射信号

| 信号 | 含义 |
|------|------|
| 问题太多 | 故事尚未准备好，先解决问题。 |
| 规则太多 | 故事太大，需要切分。 |
| 某条规则下示例太多 | 规则可能过于复杂。 |
| 规则下没有示例 | 如果所有人都理解，就不需要示例。 |

---

## 第三点五阶段：需求完整性检查

Example Mapping 擅长发现行为场景，但不会自动引导你去想异常、约束、NFR。

```
1. 主场景 — 最小可验证价值路径是否已覆盖？
2. 正常场景 — 不同合法输入组合是否已覆盖？
3. 异常场景：重复提交/并发、数据不一致、资源异常、时序异常、网络异常
4. 约束：字段输入、权限、限额、唯一性
5. UI 需求（如涉及前端）
6. 技术需求（可靠性/可扩展性/可维护性）
```

对每个发现的空白，问人类（一次一个）。人类说不需要 → 跳过。

---

## 第四阶段：将示例转换为 Gherkin

### 转换规则

1. **一个示例 = 一个场景**
2. **纯业务语言 -- 绝不使用技术术语**
3. **声明式 -- 描述"是什么"，而非"怎么做"**
4. **标记 @status-todo** -- 尚未实现
5. **多数据组合用 场景大纲 + 例子**
6. **公共前提用 背景**
7. **当 描述用户行为，那么 描述用户观测结果**
8. **场景名称必须具体**

### 业务语言检查

```gherkin
# ❌ 技术语言
当 我 POST 到 "/api/auth/login" 端点
那么 HTTP 状态码应该是 200

# ✅ 业务语言
当 我输入凭证并提交
那么 我应该看到"欢迎回来"
```

---

## 第五阶段：展示并获得确认

将 .feature 文件**逐块**展示给人类（不要一次全部展示）。

对每个功能块：
1. 展示功能描述（作为 / 我想要 / 以便）
2. 展示其下的场景
3. 询问："这是否符合你的预期？"
4. 如有需要则修改

---

## 第六阶段：写入输出文件

### 功能文件

保存到：`features/<领域>.feature`

```gherkin
功能: 用户认证
  作为 注册用户
  我想要 登录系统
  以便 访问受保护的资源

  # --- 规则: 有效凭证允许访问 ---

  @status-todo
  场景: 使用有效凭证成功登录
    假如 用户"alice"已注册且密码为"secure123"
    当 "alice"使用密码"secure123"登录
    那么 登录应该成功

  # --- 规则: 无效凭证被拒绝 ---

  @status-todo
  场景: 使用错误密码登录失败
    ...
```

规则以 Gherkin 注释形式对相关场景进行分组。

### 问题文件（如有未解决的问题）

依赖未解决问题的场景应标记 `@status-blocked` 并附带注释。

---

## 第七阶段：功能审查（子代理）

使用 `./feature-reviewer-prompt.md` 调度功能审查子代理。

审查者检查：
- 场景使用声明式业务语言
- 每个场景只测试一个行为
- 假如/当/那么 步骤可复用
- 场景之间没有矛盾
- Example Map 中的所有规则都有对应场景
- 步骤用词一致

如果审查者发现问题，修复后重新审查。最多3轮。

---

## 第八阶段：提交

```bash
git add features/
git commit -m "spec: [主题]的功能规格说明

Example Mapping: X 条规则, Y 个场景, Z 个问题
功能文件: features/xxx.feature (N 个场景)"
```

---

## 范围评估

如果需求涉及多个独立子系统：
1. 立即标记
2. 帮助分解 -- 识别依赖关系
3. 建议顺序 -- 先建基础
4. 每个子系统各自进行一次澄清功能会话

---

## 已有项目适配

如果项目已有代码但没有 .feature 文件：

1. 阅读代码库以理解现有行为
2. 为已有功能提议"补录"场景
3. 将已有行为标记为 @status-done
4. 将新行为标记为 @status-todo
5. 第一次 BDD 循环迭代验证 @status-done 场景确实通过
