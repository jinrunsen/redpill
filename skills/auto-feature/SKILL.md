# 自动设计

> 适用于小型/定义明确的需求，由 AI 自主生成 .feature 文件，
> 无需人工交互的 clarify-feature 流程。design-reviewer 子代理替代人工审批。
> 对于大型、模糊或高风险需求，请改用 clarify-feature。

---

## 何时使用本技能 vs clarify-feature

```
┌──────────────────────────────────────────────────────────────┐
│  需求到达                                                      │
│      │                                                       │
│      ▼                                                       │
│  范围检查：                                                    │
│                                                              │
│  满足以下全部条件时使用 auto-feature：                            │
│    ✅ 范围小（1 个功能，≤ 8 个场景）                              │
│    ✅ 需求具体（不是"做好一点"这种模糊描述）                       │
│    ✅ 领域已被现有代码库充分理解                                   │
│    ✅ 无需产品/业务决策（没有 🔴 类问题）                          │
│    ✅ 在 BDD 循环中运行（没有人可以询问）                          │
│                                                              │
│  满足以下任一条件时使用 clarify-feature：                         │
│    ⚠️  范围大或不明确                                           │
│    ⚠️  需求模糊（如"搭建认证系统"）                               │
│    ⚠️  领域陌生（没有现有模式可循）                                │
│    ⚠️  关键产品决策尚未确定                                      │
│    ⚠️  有人在场且可以对话                                        │
│                                                              │
│  拿不准时：选择 clarify-feature。                                │
│  自主设计无法弥补糟糕的需求。                                      │
└──────────────────────────────────────────────────────────────┘
```

---

## 激活

宣告："我正在使用 auto-feature 自主生成此需求的功能规格说明。"

本技能在以下情况激活：
- BDD 循环迭代遇到没有 .feature 文件的需求
- 任务/问题/工单需要设计但没有人在循环中
- 协调者判断范围符合自主设计条件

---

## 核心原则

**自主设计不是没有人参与的 clarify-feature。
它是从定义明确的需求中推导（ENGINEERING）规格说明。**

需求必须提供足够的信息来推导场景，而非靠猜测。
如果发现自己在猜测，立即停止——将需求标记为需要人工 clarify-feature。

---

## 流程

```
┌──────────────────────────────────────────────────────────────────────┐
│                     BDD 自动设计                                      │
│                                                                      │
│  1. 摄入     — 阅读需求 + 项目上下文                                    │
│  2. 分析     — 提取规则和示例（迷你 Example Mapping）                    │
│  3. 草拟     — 编写 .feature 文件                                      │
│  4. 自检     — 应用反模式清单                                           │
│  5. 评审     — 派遣 design-reviewer 子代理（需求契合度）                  │
│  6. 校验     — 派遣 feature-reviewer 子代理（Gherkin 质量）              │
│  7. 迭代     — 修复任一评审者提出的问题（最多 2 轮）                      │
│  8. 提交     — 保存并转入 writing-plans                                 │
│                                                                      │
│  逃生出口：在任何阶段，如果需求过于模糊，                                 │
│  立即停止并输出：NEEDS_HUMAN_DESIGN 及说明。                             │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

## 阶段 1：摄入

阅读需求的所有可用上下文：

**主要来源**（按优先级排列）：
1. **需求文档** — REQUIREMENT.md、issue 正文、工单描述、任务文本
2. **现有 .feature 文件** — 理解系统已有的行为
3. **现有代码库** — 领域模型、API 模式、命名约定
4. **AGENTS.md / CLAUDE.md** — 项目约定和约束
5. **Git 日志** — 近期变更的上下文

**构建需求理解**：

```markdown
## 需求理解

**要求做什么**：[一句话]
**谁受益**：[角色/用户画像]
**为什么重要**：[业务价值]
**已有什么**：[相关的已有功能/代码]
**可推断的约束**：[从代码库模式、AGENTS.md 推断]
**我不确定的地方**：[列表 — 如果此列表很长，触发逃生出口]
```

### 逃生检查 #1

如果"我不确定的地方"包含以下任何一项：
- 业务规则模糊（"失败 3 次还是 5 次后锁定？"）
- 范围模糊（"'用户管理'是否包括角色？"）
- 架构选择（"这应该是独立服务吗？"）
- 用户体验决策（"显示错误还是重定向？"）

则：**停止。输出 `NEEDS_HUMAN_DESIGN`。** 这些决策需要人来做。

---

## 阶段 2：分析

执行压缩版 Example Mapping（Example Mapping 即"实例映射"，将需求拆解为规则和具体示例的结构化方法） —
与交互式 clarify-feature 结构相同，但从需求文本推导而非通过对话。

### 提取规则

阅读需求并识别每条业务规则。

```markdown
## 提取的规则

1. **[规则名]**：[描述]
   来源：[需求中的哪句话/哪个段落]
   置信度：高 | 中
```

**置信度级别**：
- **高**：需求明确说明了此规则
- **中**：强烈暗示但未明确说明

如果任何规则的置信度为低 → 那是在猜测 → 逃生至人工设计。

### 为每条规则推导示例

对每条规则，推导具体场景：

| 模式 | 生成内容 |
|------|---------|
| **正常路径** | 直接的成功案例 |
| **边界值** | 有效/无效的边界（N、N+1、N-1） |
| **错误案例** | 出错的情况 |
| **状态依赖** | 不同的初始状态影响结果 |

并非每条规则都需要全部四种。简单的布尔规则 → 正常路径 + 错误案例就够了。

### 完整性扫描

推导标准场景后，对照以下类别检查。
**仅为需求文本明确提及或强烈暗示的类别生成场景。**
不要为需求未涉及的类别凭空创造场景。

```
对照需求文本检查：
□ 异常场景 — 需求是否提到了以下情况？
    □ 并发/重复操作 → 如提到，生成场景
    □ 数据不一致 → 如提到，生成场景
    □ 资源异常（不存在/已删除）→ 如提到，生成场景
□ 约束 — 需求是否指定了以下限制？
    □ 字段长度/格式 → 如指定，生成验证场景
    □ 权限限制 → 如提到角色，生成权限场景
    □ 唯一性 → 如提到"必须唯一"，生成重复场景
```

### 逃生检查 #2

如果总场景数 > 8 或规则数 > 5：范围过大，不适合自主设计。
拆分或升级。

---

## 阶段 3：草拟

将示例转化为 Gherkin。

### 草拟规则

1. **声明式而非命令式** — 描述行为，而非点击操作
2. **领域词汇** — 使用现有代码库中的术语
3. **步骤一致** — 检查现有 .feature 文件，复用相同措辞
4. **具体数值** — 真实可信的数据
5. **标记 @status-todo** — 尚未实现
6. **按规则分组** — Gherkin 注释体现规则结构

### 步骤复用检查（关键）

在编写任何新步骤之前，搜索现有的 .feature 文件和步骤定义。
新步骤的命名应遵循现有模式。

---

## 阶段 4：自检

在派遣评审者之前，验证：

```markdown
## 自检

### 格式
- [ ] 每个场景都有 @status-todo 标签
- [ ] 功能有 As a / I want / So that（作为/我想要/以便于）
- [ ] 规则以 Gherkin 注释形式记录

### BDD 质量
- [ ] 一个场景 = 一个行为
- [ ] 声明式语言（不涉及 UI/SQL/API 路径）
- [ ] 每个场景相互独立
- [ ] 无矛盾
- [ ] 步骤措辞与现有 .feature 文件一致

### 需求忠实度
- [ ] 每条提取的规则至少有一个场景
- [ ] 没有超出需求的臆造规则
- [ ] 中等置信度的规则未被当作确定的
- [ ] 范围与需求匹配（不大不小）
```

发现任何失败项先修复再继续。如果范围/模糊性检查不通过，升级处理。

---

## 阶段 5：评审 — design-reviewer 子代理

使用 `./design-reviewer-prompt.md` 派遣 **design-reviewer 子代理**。

design-reviewer 接收：
1. 原始需求文本
2. 草拟的 .feature 文件
3. 提取的规则 + 示例分析
4. 现有 .feature 文件列表（用于一致性检查）

design-reviewer 回答 feature-reviewer 无法回答的问题：
**"此设计是否忠实地表达了需求？"**

| design-reviewer（本阶段） | feature-reviewer（下一阶段） |
|--------------------------|---------------------------|
| 设计是否与需求匹配？ | Gherkin 格式是否正确？ |
| 是否遗漏了需求？ | 场景是否独立？ |
| 是否臆造了需求？ | 语言是否声明式？ |
| 范围是否正确？ | 步骤是否一致？ |
| 是否有隐含假设？ | 是否有矛盾？ |
| 是否应该由人来设计？ | — |

---

## 阶段 6：校验 — feature-reviewer 子代理

设计评审通过后，派遣 **feature-reviewer 子代理**
（来自 clarify-feature/feature-reviewer-prompt.md）检查 Gherkin 质量。

**两道关卡，而非一道。** 两者都必须通过才能继续。

---

## 阶段 7：迭代

如果任一评审者驳回：

**第 1 轮**：修复具体问题。仅重新提交给驳回的评审者。
**第 2 轮**：再次修复。再次提交。
**第 3 轮（上限）**：需求对自主设计来说过于复杂。
输出 `NEEDS_HUMAN_DESIGN`，附带：
- 草拟的 .feature 文件（人工不必从零开始）
- 评审反馈
- 自主设计遇困原因分析

---

## 阶段 8：提交与转场

### 写入文件

```
features/<领域>.feature
.redpill/decisions/YYYY-MM-DD-<主题>-auto-design.md
```

### 设计记录

```markdown
# [主题] — 自动设计记录

## 需求来源
[复制或引用原始需求]

## 设计方式
自主设计（auto-feature）— 无人工交互

## 提取的规则
1. [规则]：[来源] — 置信度 高
2. [规则]：[来源] — 置信度 高

## 生成的场景
- features/<领域>.feature：N 个场景

## 评审结果
- 设计评审：通过（第 [1|2] 轮）
- 功能评审：通过（第 [1|2] 轮）

## 做出的假设
- [中等置信度的解读 — 应由人工确认]

## 未覆盖（明确排除的范围）
- [需求未要求的内容]
```

### 提交

```bash
git add features/ .redpill/decisions/
git commit -m "spec(auto): [主题] 的功能规格说明

基于 [需求来源] 的自主设计。
规则：N 条，场景：M 个。
设计评审：通过，功能评审：通过。"
```

### 输出

```
<AUTO_DESIGN_STATUS>
status: COMPLETE | NEEDS_HUMAN_DESIGN
feature_files:
  - features/<领域>.feature（N 个场景 @status-todo）
design_record: .redpill/decisions/YYYY-MM-DD-<主题>-auto-design.md
assumptions: [列表或"无"]
next_step: writing-plans
</AUTO_DESIGN_STATUS>
```

## 防护栏

### 硬性限制

| 防护栏 | 限制 | 原因 |
|--------|------|------|
| 最大场景数 | 8 | 更大范围需要人工 |
| 最大规则数 | 5 | 规则越多风险越大 |
| 最大评审轮数 | 2 | 2 轮都过不了 = 升级 |
| 置信度门槛 | 全部为高或中 | 低 = 猜测 = 升级 |
| 最大假设数 | 2 个中等置信度 | 更多 = 猜测过多 |

### 红旗 → 立即升级为 NEEDS_HUMAN_DESIGN

- 需求在不同方案之间用"或"连接
- 需求引用了未定义的术语
- 需求与现有 .feature 文件矛盾
- 多个用户角色有不同权限
- 需求说"类似于 X"但 X 不在代码库中
- 任何一句话需要读三遍才能理解
- 需求只是一句模糊的话，没有具体细节
